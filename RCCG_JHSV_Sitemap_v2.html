RCCG Jesus House Silicon Valley

Sitemap & Information Architecture Blueprint · Stakeholder Review Draft · April 2026

This document translates the design review into a concrete, functional sitemap. It focuses on structure, content inventory, and button placement — not visual design. Wireframes are intentionally low-fidelity so the team can debate what goes where and why, before any pixels are pushed.

The core problem we are solving: today's site has 20 pages, six fonts, a hidden hamburger nav, and four decorative buttons that lead nowhere informative. The new site consolidates to 8 primary pages, makes the path to "Plan Your Visit" the most prominent action on every screen, and treats the site as the single source of truth for service times, sermons, giving, media, and social — with everything discoverable in ≤2 clicks.

What this sitemap improves vs. the current site

Discoverability

Desktop gets a visible horizontal nav. No more hamburger-only navigation hiding 9 pages on a 1920px monitor.

Single conversion goal

Every page funnels toward /new (Plan Your Visit) or /live. One hero, one CTA.

Media as a hub

Livestream, gallery, podcast, and social all reachable from the top nav or one click into it. Sermons phased to v2 with YouTube as the interim destination.

Mobile parity

Breakpoints for 320–767 (mobile), 768–1023 (tablet), 1024+ (desktop). Touch targets ≥44px.

SEO foundation

Church schema, per-page meta, Google Business Profile, and renamed /blog → /sermons.

Footer as reference, not a map

Footer shrinks from 1,529px to ~320px. Becomes a quick-reference card, not a second navigation.

Sermons PHASE 2

/sermons
Launch strategy — phased. v1: the "Sermons" nav item links directly to the church's YouTube channel (opens in new tab). No internal page built at launch. v2: replace the external link with a full /sermons page powered by the YouTube Data API v3 — videos flow in automatically, organized by series, with on-site player and podcast feed.

v1 behavior at launch (weeks 1–8)

Nav item: "Sermons"
Nav link href="https://www.youtube.com/@[channel-handle]" with target="_blank" rel="noopener".
Destination: the church's YouTube channel page.
User experience: click "Sermons" in nav → new tab opens YouTube → user browses the channel's Videos tab, Live tab, or Playlists.
Sermons ↗ → opens YouTube channel in new tab
Why defer the page: a /sermons archive only feels full if it's kept fed. Shipping an empty page before the weekly content cadence is proven creates a worse first impression than routing visitors to YouTube — which is already a polished archive of the church's full sermon library. YouTube also has search, captions, and playback speed controls built in.
Required before v1 launch: verify the YouTube channel has (a) a recognizable handle/URL, (b) a custom banner and logo, (c) at least the most recent 6 sermons uploaded as public videos, and (d) one "Sermons" playlist that pins or highlights current content. Without these, the YouTube tab will look neglected.

v2 page design (post-launch build)

Full /sermons page architecture. Implementation triggered when weekly content ops are stable (typically month 3+).

0 · YouTube Data API Integration
Every page load (or cached, hourly) the page calls youtube.v3.search.list / playlistItems.list against the church's channel ID to pull:
  • The 12 most recent uploads (for "Latest" and list view)
  • Playlists tagged as sermon series (for "All Series" grid)
  • Live broadcast status (to surface a banner if currently streaming)
Data cached server-side (or via Wix Velo backend) to stay within API quota (10,000 units/day default). No manual blog posting required — every new YouTube upload appears on the site automatically.
Why YouTube API, not manual posts: the original plan assumed weekly manual sermon blog entries with 3-paragraph summaries. That's a real ongoing content lift. YouTube API integration cuts the weekly ops to zero — upload to YouTube, it appears on the site. Summaries/SEO content can be layered on later if desired.

Watch Live

/live
Primary goal: zero-friction way to join the service online. The page always shows something playable — never a blank "offline" dead-end. Powered by the YouTube Data API to auto-detect whether the channel is live, scheduled, or offline.
0 · YouTube Data API integration
On page load, the page calls youtube.v3.search.list against the church's channel ID with three queries to determine state:
  • eventType=live — is there a broadcast happening right now?
  • eventType=upcoming — is one scheduled within the next 24 hours?
  • eventType=completed — most recent past broadcast (fallback)
The first result that returns a video ID drives the embed. Every "Watch Live" click in the nav lands on a working player — no stale or broken stream URLs to maintain manually.
Implementation detail: API key stored in a Wix Velo backend function (or lightweight Node proxy), not exposed client-side. Results cached for 60 seconds to stay within the 10,000/day quota and avoid rate-limit issues during peak Sunday traffic.
1 · Live Status Indicator (dynamic, driven by API result)
Badge state switches based on the API response:
  • 🔴 LIVE NOW — when eventType=live returns a broadcast
  • Upcoming: Sun 10:30 AM PT — when a stream is scheduled within 24h
  • Most recent service — when offline; player shows the latest completed broadcast
2 · Primary Stream Embed (16:9, full-width)
YouTube iframe embed at youtube.com/embed/[video-id], where the video ID comes from the API call above. If the API fails entirely, fallback to a hard-coded latest-upload embed so the page never renders empty.
3 · Service Schedule + Add-to-Calendar
Sunday · 10:30 AM PT · Wednesday · 7:00 PM PT. "Add to Calendar" button downloads an ICS file so it lands on the viewer's device regardless of calendar app.
4 · During-Service Actions (right rail on desktop, stacked on mobile)
• Give — quick tap during offering
• Prayer Request
• Connection Card
• Links: Facebook Live mirror · YouTube chat
Why right-rail actions matter: a remote viewer who wants to give or submit a prayer request during service must not have to leave the stream. Peer pattern from Echo and Union.
5 · Past Services
Row of last 3 broadcasts (pulled from same API call with eventType=completed). Each links to the YouTube video, or in v2 routes to the internal /sermons/[id] page.

5. Button & CTA Placement Principles

Rules the team agrees on upfront so every page gets the same treatment.

PrincipleRuleWhy
One hero, one CTA Every hero has exactly one primary CTA. "Watch Live" in the nav bar is the escape hatch for the other audience. Two hero buttons halve conversion on each. Peer sites (Echo, Union, Well) all converged on a single hero CTA.
Persistent Watch Live Purple filled button, far right of nav, sticky on scroll, visible on every page. Second-most-common visitor intent after "plan a visit." Needs zero-click access.
Primary vs. Outlined Primary filled = one per section max. Outlined = everything secondary. Ghost/text = tertiary. Button hierarchy has to be visual, not accidental.
CTAs belong near content that earns them "Learn More" lives under the paragraph that introduced what to learn about. No floating orphan buttons. A button without context is a gamble the visitor won't take (Issue #5 in the review).
Mobile: full-width buttons All primary buttons go to 100% width on mobile with 24px side padding. Touch targets + thumb reach. Echo, Well, and Union all do this.
Pre-footer CTA band on every page Purple band above the footer with the page's most relevant next action. Catches scrollers who didn't convert higher up. Last chance before they leave.
External links open in new tab Google Maps, YouTube, Facebook, Instagram, giving platform — target="_blank" rel="noopener". Don't lose the visitor to a back-button round-trip.
No hover-only affordances Tap targets must be obvious without hover. iPad + touch laptops have no hover. Accessibility + modern device reality.

Stakeholder review questions

  1. Do we agree the primary nav is Home · I'm New · About · Sermons · Events · Give + Watch Live — and that "Ministries" lives in the mobile menu and footer but not desktop top nav?
  2. Do we agree Sermons is phased to v2, with the nav link pointing directly to YouTube at launch? (See Sermons page blueprint for required channel hygiene before v1 ships.)
  3. Who owns the YouTube channel? Do we have the handle/URL, channel ID, and access to Google Cloud Console for issuing an API key?
  4. Do we keep Gallery as a section inside /about + full archive in /media, or should Gallery be promoted to the top nav?
  5. Is "I'm New" the right label, or do we prefer "Plan Your Visit" / "Visit"? (Shorter labels scan better on mobile.)
  6. Who owns the connection card inbox and the 3-email welcome sequence?
  7. Which giving platform? (Tithe.ly, Pushpay, Stripe, or keeping current Wix-embedded solution?)
  8. Do we launch a Media Hub (/media) at v1, or defer to v2 alongside /sermons?
  9. What's the trigger to start v2 work? (E.g., "once we've published 12 consecutive weeks of sermons to YouTube with clean metadata," or a date-based commit.)
Recommendation — v1 launch scope: ship 7 built pages (Home, I'm New, About, Events, Give, Live, Contact) plus a Sermons nav link that redirects to the church's YouTube channel. Watch Live uses the YouTube Data API to surface the current broadcast state. Defer /sermons, /ministries, and /media to v2 — trigger the v2 build once the YouTube channel has 12 weeks of consistent uploads and the weekly content ops have stabilized.
Why this scope works: the highest-value pages for a first-time visitor (/new, Home, service times, giving, live service) all ship at v1. The Sermons-via-YouTube interim gives visitors access to the full sermon archive from day one — YouTube is already a polished player with search, captions, and mobile apps. The team avoids building hubs they can't feed, and the v2 build becomes easier because the YouTube channel is already the canonical source of truth.
